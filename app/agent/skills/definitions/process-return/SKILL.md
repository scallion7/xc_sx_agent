## 退货退款处理流程

当用户表达退货/退款/换货意愿时，按以下流程处理。

### 第一步：确认订单信息

- 如果用户提供了订单号，调用 `query_order` 查询订单详情
- 如果用户没有提供订单号，调用 `list_user_orders` 列出用户所有订单，请用户确认是哪个订单
- 确认后，向用户复述订单关键信息（商品名称、金额、下单时间），请用户确认

### 第二步：判断订单状态

根据 `query_order` 返回的 status 字段判断：

- **pending（待发货）**：可以直接取消订单并退款，告知用户无需退货，退款 1-3 个工作日原路退回
- **shipped（已发货）**：需要等签收后再申请退货，或者可以选择拒收，拒收后 5-7 个工作日自动退款
- **delivered（已签收）**：进入第三步检查退货资格
- **refund_processing（退款处理中）**：告知用户已有退款申请正在处理，请耐心等待，无需重复申请

### 第三步：检查退货资格

调用 `search_knowledge` 检索退换货政策，确认以下要点：

1. **时效**：签收后是否在 7 天内（七天无理由）或 15 天内（质量问题）
2. **商品类型**：是否属于不支持退货的类别（定制商品、鲜活易腐、数字商品、拆封贴身衣物）
3. **退货原因**：
   - 七天无理由：运费由用户承担（12 元起，从退款中扣除）
   - 质量问题：运费由平台承担，需要用户提供照片/视频举证

向用户说明退货资格和运费承担情况。

### 第四步：确认意愿并创建售后工单

- 向用户复述订单号、商品、售后类型和原因，取得明确确认（不能跳过）
- 确认后调用 `create_after_sale_case`，将 `user_confirmed` 设为 true
- 质量、破损、错漏发场景如已提供证据，将 `evidence_provided` 设为 true
- 根据返回的风险路由处理：
  - `auto_process` / `approved`：可以调用 `execute_after_sale_case`
  - `manual_review` / `human_escalation`：告知工单号和待审状态，不得继续执行

### 第五步：Human-in-the-Loop 审批

- 人工审批通过后，工单状态变为 `approved`，此时才能执行
- 人工拒绝后，说明拒绝原因并提供补充材料或申诉渠道
- Agent 无审批工具，禁止声称自己已经批准工单
- 用户询问进度时调用 `query_after_sale_case`，不得凭对话猜测

### 第六步：告知后续流程

根据退款结果，告知用户：
- 预计退款到账时间（1-3 个工作日）
- 如果需要寄回商品，提示等待退货地址通知
- 主动询问用户是否还有其他问题

### 注意事项

- 退款是敏感操作，申请前必须与用户确认订单号和退款原因，不能替用户做决定
- 禁止直接调用 `apply_refund` 绕过售后工单、风险路由和人工审批
- 政策结论必须引用 `search_knowledge` 返回的 citation；`grounded=false` 时转人工
- 如果用户是钻石会员，提醒其可使用"无忧退"权益（一年 4 次，无需理由，运费全免）
- 如果用户情绪激动，先安抚情绪再处理问题
- 遇到复杂情况（如部分退款、多件商品退其中一件），建议转人工客服
