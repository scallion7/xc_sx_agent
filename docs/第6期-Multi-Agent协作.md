## 选哪种 Multi-Agent 架构

Multi-Agent 架构有很多种。中心化调度、去中心化协商、DAG 有向图、甚至带环的循环图（Agent A 的输出可以回流给 Agent B 再回来）。架构越复杂越灵活，但也越难调试、越容易出现死循环或不可预测行为

我们的场景是电商客服，用户每轮消息基本对应一个明确意图：问商品、查物流、投诉。不需要多个 Agent 来回讨论形成共识，也不需要 DAG 编排多步依赖任务

所以选了最朴素的：Router 做意图识别，分发给平级的专家 Agent（售前/售后/投诉）一次性处理完。没有环、没有 Agent 间互调，一条直线走到底。够用就是最好的架构



## Agent 间怎么通信

多个 Agent 之间要共享上下文，常见做法是像 LangGraph 那样搞一个 State 对象，每个 Node 从 State 里读需要的变量、处理完再写回去，通过 State 的流转实现 Agent 间通信

我们没引 LangGraph，用了更直接的方式：Orchestrator 持有一份共享对话历史，所有子 Agent 执行时都能看到完整上下文。用户先问售前"这款手机拍照怎么样"，再转售后"帮我退了吧"，售后 Agent 天然知道退的是哪个商品，不需要额外的变量传递

本质区别在于：State 传递适合 Agent 间有复杂数据依赖的场景（比如研究任务链），而共享对话历史适合"同一个用户、多轮交互、切换专家"的场景。客服刚好是后者
