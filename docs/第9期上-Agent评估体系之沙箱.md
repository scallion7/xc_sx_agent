高级篇最后一期：Agent 评估体系。这块内容含金量很高，我拆成两篇。这是上篇，先解决一个最容易被忽略的前提——沙箱



前八期每次改完代码，到底变好还是变差，我都是手动跑两句凭感觉。这在面试里是要扣分的，正经的做法得能量化。但量化的第一步不是急着定指标，而是先有一个能稳定复现的运行环境



1️⃣ 为什么评估非要先搭沙箱

Agent 身上有两个会偷偷污染结果的东西：记忆和会话历史

同一条用例跑两次，如果上一次的长期记忆被读进了 prompt，这一次的表现就被带跑偏了，结果没法比。这里有个很多人会踩的坑：你以为「不触发记忆写入」就干净了，其实记忆在初始化时就会被读进上下文，光不写没用，得显式关掉

还有工具。真实工具的返回可能随时间变化，评估时必须换成固定的 mock，这样每次的事实基准都一样。把记忆、历史、工具这三个变量都钉死，Agent 才进入一个可复现的真空环境



2️⃣ 沙箱怎么把运行过程录下来

光能跑还不够，评估要的是过程数据：这一轮烧了多少 token、调了哪些工具、按什么顺序调的、最后回了什么

我用的办法是给 Agent 调用大模型的那个入口做插桩，整场对话每一次模型调用都被记下来。巧的地方在于，不管是单 Agent 还是多 Agent 协作，底层都共用同一个调用入口，所以插一个点就能把全链路的轨迹收齐，Agent 本身的代码一行都不用动

跑完一条用例，就得到一份完整的 trace，里面装着过程和结果。下篇的打分，全靠它
