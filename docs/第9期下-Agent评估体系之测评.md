Agent 评估体系下篇。上篇搭好了沙箱，跑完一条用例能拿到一份 trace。这篇讲怎么把这份 trace 变成分数



很多人评估 Agent 只看最后那句答得对不对。但这会漏掉一大类问题：答案是对的，可它绕了一大圈、调错了工具、烧了三倍 token。这种东西上了生产是要出事的，所以我把指标分成两层



1️⃣ 过程指标和结果指标

过程指标看 Agent 怎么干活的：该调的工具有没有调对、调用次数是不是接近理论最少、token 有没有超预算

结果指标看最后交付得怎么样：意图识别准不准、关键信息全不全、该不该转人工、回答质量、有没有幻觉

只看结果会放过一个绕远路的 Agent，只看过程又抓不住答非所问。两层都测，才能既要效率又要质量



2️⃣ 工程规则判断 vs LLM judge

不是所有指标都得用大模型打分。token 消耗、调用次数、关键词命中这些，写几行代码就能算，确定、免费、可复现，能用规则的就别浪费一次模型调用

但有些东西规则真写不出来。回答质量好不好、有没有编造工具根本没返回的信息，这需要「读懂语义」才能判，只能交给 LLM judge

幻觉检测我觉得最有意思：把工具的真实返回当成唯一可信的事实，喂给裁判模型，让它逐条核对回复里的每个具体事实有没有依据。我实测时 Agent 把顺丰写成了圆通，立刻就被抓出来了

跑下来还故意留了几个不及格的用例。比如知识库没建索引时 Agent 编了段政策，幻觉这关直接挂掉——这恰恰说明评估在干活。靠放水刷到满分没意义，能解释清楚为什么挂，才是评估的价值
