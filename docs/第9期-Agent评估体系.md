前面八期小夕能查订单、调工具、检索知识库、多 Agent 协作、记忆、用技能了。但每次改完代码，到底是变好了还是变差了？我一直靠手动跑 `main.py` 聊两句凭感觉，这显然不靠谱

这期给项目加 Agent 评估体系，聊两个核心问题



## 怎么让 Agent 评估可复现

评估的前提是「同样的输入，每次跑出来的过程和结果可比」。但小夕身上有两个会偷偷污染结果的东西：记忆和会话历史

我搭了一个沙箱（Sandbox）来隔离。每条用例用独立的临时 session 文件，互不串台；关掉记忆——注意这里有个坑，光「不调 close」只挡住了长期记忆的写入，`MemoryManager` 初始化时还会 `ltm.load()` 把 `default.json` 读进来注入 prompt，所以必须显式把 `memory_enabled` 关掉；再关掉 MCP 只用本地 mock 工具，保证工具返回是确定的

沙箱还顺手把运行过程「录」下来。关键技巧是：单 Agent 和多 Agent 其实**共享同一个 OpenAI client 实例**，所以我只要给这个 client 的方法打个补丁（monkeypatch），就能捕获整场对话每一次 LLM 调用的 token、调了哪些工具、耗时多久，全程不用改 `chat.py` 一行代码。工具的返回值则通过包裹 `ToolManager.execute_tool` 收集起来，留着做幻觉检测的对照证据



## 过程指标和结果指标，为什么都要测

很多人评估 Agent 只看最后那句回答对不对。但这会漏掉一大类问题：答案是对的，可它绕了一大圈、调错了工具、烧了三倍 token——这种在生产环境是要出事的

所以我分了两层。**过程指标**看 Agent 怎么干活的：工具调用准不准、调用次数是不是接近理论最少、token 有没有超预算。**结果指标**看最后交付得怎么样：意图识别、关键信息完整性、是否该转人工、回答质量、有没有幻觉

这两层里，有的指标用代码规则就能判——token 消耗、调用次数、关键词命中，确定性强、零成本、可复现；有的必须上 LLM judge——回答质量好不好、有没有编造工具没返回的信息，这些需要「理解语义」，规则写不出来。幻觉检测尤其有意思：我把工具的真实返回喂给裁判模型当 ground truth，让它核对回复里的每个具体事实有没有依据

面试被问到「你怎么评估 Agent」，能讲清楚「先有可复现的沙箱，再分过程/结果两层、代码规则与 LLM judge 各管一摊」，比只说「我跑了几个 case 看着还行」要有说服力得多
