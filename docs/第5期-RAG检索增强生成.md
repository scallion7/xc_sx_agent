## 为什么需要 RAG

把政策全塞 prompt 里不行吗？三个问题：token 贵（每轮都带几万字）、召回差（上下文太长注意力稀释）、更新难（改政策要发版）

RAG 的思路：知识库离线切片→向量化→存索引。用户提问时，问题也向量化，余弦相似度找 Top-K 片段塞给模型。省 token、命中准、改文档不用动代码



## RAG First vs Agentic RAG

大多数 RAG 教程的做法是 **RAG First**：用户一说话，先无脑检索知识库，把结果塞进 prompt 再让模型回答。简单直接，但有个根本问题——不是所有问题都需要检索。用户说"帮我查订单 ORD-001"，这时候该调工具不该翻知识库，强行检索只会引入噪音、浪费 token

我们这期做的是 **Agentic RAG**：把知识库检索包装成一个工具 `search_knowledge`，注册到 Agent 的工具表里，由模型在 ReAct 循环中自主判断要不要调。问政策→检索，查订单→调工具，混合问题→两个都调。这正是 Agent 的价值——让模型自己决策而不是人写 if-else

面试如果被问到"你的 RAG 是怎么触发的"，能答出这两种范式的区别，比只说"我用了 LangChain"要强得多



## 两套向量后端

做了两套实现，一个学原理一个学工程：

- **NumpyBackend**：手写余弦 O(n) + JSON 持久化，不到 100 行，教学透明
- **ChromaBackend**：嵌入式向量数据库，HNSW 索引 O(log n)，生产标配

为啥生产都用向量数据库？HNSW 亿级规模仍是毫秒级；元数据过滤、增量更新、多租户隔离是刚需；还自带查询监控。但教学的意义在于：先看清"本质就是余弦 + Top-K"，再去理解 HNSW 怎么优化的

`RAG_BACKEND=numpy` 或 `chroma` 一行切换，构建时 `--backend` 参数显式指定
